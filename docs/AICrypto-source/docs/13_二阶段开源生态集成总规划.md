# AICrypto 二阶段开源生态集成总规划

## 1. 文档定位

本规划站在 AICrypto 总设计师视角，回答一个更高阶的问题：

在已经完成首轮架构设计、首轮开源调研、实时联网核验和第一批仓库归档之后，AICrypto 接下来 3-6 个月应该如何系统化地吸收开源生态，并把这些能力真正转化为平台能力。

本阶段不是继续“堆仓库”，而是进入“平台化吸收”的第二阶段。

## 2. 二阶段的核心目标

二阶段的重点不是新增多少代码，而是建立四种平台能力：

1. 吸收能力
2. 协议化能力
3. 模块化能力
4. 进化能力

具体目标包括：

- 把第一批高价值仓库从“归档研究”推进到“模块化嫁接设计”
- 把第二批观察池建立起来，形成持续补充机制
- 把 Binance 官方新 connector 体系与 AICrypto 自建 Adapter 路线理顺
- 把研究层、知识层、数据层、执行层的开源接入边界彻底明确
- 为后续正式工程初始化和模块落地提供路线图

## 3. 二阶段总体判断

经过这几轮实时联网调研和本地仓库核对，可以做出一个明确判断：

- AICrypto 不缺“可参考仓库”
- 真正稀缺的是“如何把不同仓库变成统一平台能力”

因此二阶段最重要的事情是：

- 不要继续做一堆离散调研
- 要开始建立“平台兼容层”和“模块导入制度”

## 4. 二阶段能力地图

## 4.1 研究层

目标：

- 快速提升策略研究和回测效率

主力来源：

- `freqtrade`
- `Riskfolio-Lib`
- `TradingAgents`
- `LangGraph`

二阶段动作：

- 抽取研究流程模板
- 统一 Strategy Spec / Skill Spec
- 统一研究报告结构
- 建立策略研究工坊标准流程

关键产出：

- `research workflow`
- `strategy experiment template`
- `backtest evidence template`
- `skill proposal template`

## 4.2 数据层

目标：

- 提升 Binance Futures 数据接入、标准化、存储和回补能力

主力来源：

- `cryptofeed`
- `exchanges-lab/view`
- Binance 官方 modular connector

二阶段动作：

- 定义正式 Canonical Event 标准
- 建立市场数据采集沙箱
- 建立实时流 + 历史回补 + 数据校验三位一体机制

关键产出：

- `market ingest adapter contract`
- `canonical event schema`
- `historical backfill policy`
- `data quality checklist`

## 4.3 知识层

目标：

- 把文档、研究结果、回测报告、Skill 资产沉淀成平台知识能力

主力来源：

- `llama_index`
- 后续 Graph / GraphRAG 生态

二阶段动作：

- 建立 Skill 文档库
- 建立复盘知识库
- 建立研究结论索引
- 建立事件档案检索层

关键产出：

- `knowledge object model`
- `skill memory model`
- `review archive model`
- `research rag topology`

## 4.4 Agent 层

目标：

- 形成可控、多角色、可持久化的 Agent 工作流

主力来源：

- `langgraph`
- `TradingAgents`
- 第二批观察池中的 Agentic orchestration 项目

二阶段动作：

- 明确 Agent 角色边界
- 明确 Agent 可调用工具边界
- 明确 Agent 不能直接绕过风控和执行网关
- 建立 Research / Debate / Review 三类图式流程

关键产出：

- `agent role contract`
- `agent graph design`
- `human approval checkpoints`
- `agent audit trail`

## 4.5 执行层

目标：

- 为未来正式实盘执行层建立长期路线

主力来源：

- `nautilus_trader`
- `ai-auto-trader-ahh`
- Binance 官方 connector

二阶段动作：

- 吸收 order state machine 设计
- 吸收 execution semantics 和 risk veto 模式
- 区分正式执行内核与研究/旁路执行器

关键产出：

- `execution runtime contract`
- `risk gateway contract`
- `order lifecycle model`
- `adapter boundary policy`

## 7. 二阶段架构纪律

### 7.1 不允许的误区

- 直接把外部框架当作主平台
- 把研究层项目直接接实盘
- 把 Agent 直接赋予真实下单控制权
- 让开源项目的数据结构反向定义 AICrypto 内核

### 7.2 必须坚持的纪律

- 所有外部数据先标准化
- 所有外部策略先协议化
- 所有 Agent 输出先过 Risk Gateway
- 所有引入模块必须可替换、可禁用、可审计

## 8. 二阶段组织方式

建议按四个工作组推进：

### A. Open Source Radar 组

负责：

- 持续发现项目
- 评级
- 更新观察池

### B. Protocol & Architecture 组

负责：

- 协议冻结
- 模块边界
- 兼容层设计

### C. Sandbox 组

负责：

- 验证开源仓库真实可用性
- 评估嵌入成本

### D. Integration Design 组

负责：

- 将通过验证的能力设计为正式嫁接模块

## 9. 二阶段最重要的设计结论

### 结论 1

研究层和知识层可以优先借助开源生态快速起势。

### 结论 2

执行层和风控层必须长期掌握在 AICrypto 内核手里。

### 结论 3

平台真正的护城河不是“某个开源仓库”，而是：

- 协议
- 风控
- 审计
- 模块治理
- 持续吸收能力

## 10. 最终建议

二阶段应当把工作重心放在：

1. 开源仓库能力地图
2. 平台协议地图
3. 模块嫁接地图
4. 3-6 个月执行路线图

AICrypto 下一阶段不应该继续停留在“调研很多项目”，而要正式进入“把项目能力平台化”的阶段。
