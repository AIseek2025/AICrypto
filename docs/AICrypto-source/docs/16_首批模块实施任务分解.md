# AICrypto 首批模块实施任务分解

## 1. 文档目标

本文件将总纲、架构、开源调研、嫁接规划进一步落实为首批可执行任务。

目标不是立刻编码，而是明确：

- 第一批真正要启动哪些模块
- 每个模块的目标、边界、输入、输出是什么
- 先后顺序是什么
- 哪些任务必须串行，哪些可以并行

## 2. 首批实施总策略

首批实施坚持三条原则：

- 先协议，后实现
- 先研究/数据/知识，后执行实盘
- 先沙箱，后正式模块

因此第一批实施优先级如下：

1. 协议层
2. 数据层
3. 研究层
4. 知识层
5. Agent 层
6. 风控层
7. 执行层设计壳

## 3. 模块拆解

## 模块 A：协议层基础包

目标：

- 定义 AICrypto 最稳定的核心协议

子任务：

1. 定义 `CanonicalEvent`
2. 定义 `FeatureVector`
3. 定义 `SkillSpec`
4. 定义 `StrategySpec`
5. 定义 `OrderIntent`
6. 定义 `RiskDecision`
7. 定义 `ExecutionReport`

输入：

- 现有架构文档
- 币安 API 语义
- 开源框架对象模型参考

输出：

- 协议定义文档
- 字段标准
- 版本规则

优先级：

- `P0`

依赖：

- 无

## 模块 B：Binance 市场数据 Adapter 沙箱

目标：

- 建立面向 Binance Futures 的标准化市场数据接入沙箱

子任务：

1. 识别必须接入的 feed
2. 设计 raw event 到 canonical event 的映射
3. 设计历史回补机制
4. 设计数据质量检查清单
5. 设计 replay 输入格式

参考来源：

- `cryptofeed`
- Binance 官方 connector
- `exchanges-lab/view`

输出：

- 数据接入设计文档
- feed 列表
- 映射规则
- 回补策略

优先级：

- `P0`

依赖：

- 模块 A

## 模块 C：研究工坊与回测沙箱

目标：

- 形成首个研究与回测工作台规范

子任务：

1. 定义策略实验模板
2. 定义回测证据模板
3. 定义参数搜索模板
4. 定义研究报告模板
5. 定义策略进入 Skill 化流程

参考来源：

- `freqtrade`
- `Riskfolio-Lib`

输出：

- 研究流程说明
- 回测产出规范
- 策略验收门槛

优先级：

- `P0`

依赖：

- 模块 A
- 模块 B

## 模块 D：Skill Registry 设计

目标：

- 把研究成果转成标准 Skill 资产

子任务：

1. 定义 Skill 元数据
2. 定义 Skill 输入输出协议
3. 定义 Skill 状态流转
4. 定义 Skill 版本与失效规则
5. 定义 Skill 与策略、风控、Agent 的关系

输出：

- Skill Registry 设计文档
- Skill 生命周期
- Skill 分类体系

优先级：

- `P0`

依赖：

- 模块 A
- 模块 C

## 模块 E：知识层与研究记忆库

目标：

- 把研究报告、复盘结论、Skill 文档沉淀为可检索知识

子任务：

1. 定义 Knowledge Object
2. 定义文档分层
3. 定义索引与检索入口
4. 定义 Skill Memory 结构
5. 定义复盘归档结构

参考来源：

- `llama_index`

输出：

- Knowledge 模型
- 文档索引规范
- RAG 拓扑说明

优先级：

- `P1`

依赖：

- 模块 A
- 模块 C
- 模块 D

## 模块 F：Research / Debate / Review Agent 图设计

目标：

- 为未来 Agent 工作流建立受控图式流程

子任务：

1. 定义 Agent 角色
2. 定义每个 Agent 的工具边界
3. 定义审批检查点
4. 定义 Agent 状态持久化需求
5. 定义 Agent 审计轨迹

参考来源：

- `langgraph`
- `TradingAgents`

输出：

- Agent 图设计文档
- Role Contract
- 审批点列表

优先级：

- `P1`

依赖：

- 模块 D
- 模块 E

## 模块 G：Risk Gateway 设计

目标：

- 建立统一风险闸门

子任务：

1. 定义 pre-trade 风控
2. 定义 in-trade 风控
3. 定义 post-trade 风控
4. 定义连亏、限仓、相关性暴露、熔断规则
5. 定义 Agent / Skill 输出进入风控的流程

参考来源：

- `ai-auto-trader-ahh`
- `nautilus_trader`
- 前期风控规划文档

输出：

- Risk Gateway 设计文档
- 风控规则清单
- veto 机制说明

优先级：

- `P1`

依赖：

- 模块 A
- 模块 D
- 模块 F

## 模块 H：Execution Runtime 设计壳

目标：

- 为后续测试网和实盘执行打下正式架构基础

子任务：

1. 定义订单生命周期
2. 定义订单状态机
3. 定义对账流程
4. 定义异常补偿
5. 定义 kill switch 机制

参考来源：

- `nautilus_trader`
- Binance Futures API 设计

输出：

- 执行层设计文档
- 状态机图
- 对账与补偿说明

优先级：

- `P1`

依赖：

- 模块 A
- 模块 G

## 4. 实施顺序

建议严格按以下顺序推进：

1. 模块 A
2. 模块 B
3. 模块 C
4. 模块 D
5. 模块 E
6. 模块 F
7. 模块 G
8. 模块 H

## 5. 并行策略

可并行工作：

- `模块 B` 与 `模块 C` 可在协议层完成后并行推进
- `模块 E` 与 `模块 F` 可在 Skill Registry 清晰后并行推进

不宜并行过早推进的工作：

- Risk Gateway 不应早于 Skill 体系
- Execution Runtime 不应早于 Risk Gateway 设计

## 6. 交付件清单

第一轮应交付的不是代码，而是以下正式资产：

- 协议文档
- 映射文档
- 模块说明书
- 状态机图
- 审批点设计
- 风控规则清单
- 目录规划

## 7. 验收标准

每个模块在进入下一阶段前至少要满足：

- 边界清晰
- 输入输出明确
- 与平台协议一致
- 能映射到目录结构
- 能映射到后续正式模块

## 8. 角色分工建议

### 总设计师

负责：

- 冻结边界
- 冻结协议
- 冻结优先级

### 架构设计

负责：

- 模块接口
- 数据流
- 状态机

### 研究工程

负责：

- 回测流程
- 研究工坊
- Skill 化规则

### Agent / Knowledge 工程

负责：

- Agent 图
- RAG 拓扑
- 文档归档

### 风控 / 执行工程

负责：

- 风控规则
- 执行状态机
- 对账与补偿

## 9. 一句话结论

首批实施不是“马上写很多业务代码”，而是：

- 先搭骨架
- 先定协议
- 先定边界
- 先让未来每一个模块都有地方可落、有标准可接

只要这一步做对，后面正式进入工程初始化时，AICrypto 就不会乱。
