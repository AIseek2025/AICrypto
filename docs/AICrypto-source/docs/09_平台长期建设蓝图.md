# AICrypto 平台长期建设蓝图

## 1. 蓝图定位

AICrypto 的长期目标不是做一个短命的量化脚本集合，而是建立一个可持续吸收开源能力、持续沉淀策略知识、持续升级自动化水平的平台。

这个平台必须同时具备：

- 长期可扩展
- 高模块化
- 高可插拔
- 强风控约束
- 强审计与可追溯

## 2. 长期目标

### 2.1 市场维度

从单一 Binance USDⓈ-M 永续起步，逐步支持：

- Binance COIN-M
- Bybit / OKX / Hyperliquid
- CEX + DEX
- 现货 + 永续 + 期权

### 2.2 智能维度

从规则策略和研究驱动 Skill，逐步演进到：

- 事件理解
- 图谱关联推理
- 多 Agent 协作
- 研究到复盘闭环自我增强

### 2.3 平台维度

从单仓库工程演进到：

- 平台核心
- 插件系统
- Skill 市场
- Data Connector 市场
- Research / Report / Agent 模板市场

## 3. 长期架构原则

## 3.1 内核最小化

长期稳定、尽量少变的只有这些内核对象：

- Canonical Event
- Feature Vector
- Strategy Spec
- Skill Spec
- Risk Policy Spec
- Order Intent
- Order State
- Audit Trace

其他内容都尽量做成可替换模块。

## 3.2 核心内生化

必须掌握在 AICrypto 自己手里的能力：

- 正式交易所 Adapter
- Risk Engine
- Execution Router
- Order State Machine
- Position / PnL Engine
- 审计与权限体系

这些能力不能长期依赖外部项目。

## 3.3 外围插件化

适合长期插件化的能力：

- 研究工具
- 回测工具
- 特征包
- Skill 包
- Agent 包
- RAG 层
- 图谱层
- 报告层
- 外部事件源

## 4. 分层蓝图

## 4.1 Foundation Layer

职责：

- 配置管理
- 密钥管理
- 日志
- 监控
- 追踪
- 任务调度
- 消息总线

要求：

- 与业务解耦
- 可替换
- 多环境统一

## 4.2 Protocol Layer

职责：

- 定义平台协议
- 屏蔽外部仓库差异

关键协议：

- `CanonicalEvent`
- `MarketSnapshot`
- `FeatureVector`
- `SignalEvent`
- `SkillSpec`
- `OrderIntent`
- `OrderExecutionReport`
- `RiskDecision`
- `ReviewReport`

原则：

- 新开源模块先适配协议
- 禁止外部项目直接定义核心对象

## 4.3 Adapter Layer

职责：

- 与外部系统连接

类型：

- Exchange Adapter
- Social Adapter
- News Adapter
- Onchain Adapter
- Knowledge Adapter
- Model Adapter

长期要求：

- 版本化
- 可插拔
- 可隔离测试
- 可灰度发布

## 4.4 Intelligence Layer

职责：

- 研究
- 特征
- 状态判断
- 图谱
- RAG
- Agent 编排

核心模块：

- Feature Engine
- Market State Engine
- Correlation Engine
- Graph Engine
- Knowledge Engine
- Agent Orchestrator
- Backtest Engine
- Portfolio Lab

## 4.5 Decision Layer

职责：

- 把洞察转成受约束的候选交易动作

核心模块：

- Skill Router
- Capital Allocator
- Portfolio Optimizer
- Execution Planner
- Risk Gateway

要求：

- 可解释
- 可复现
- 可审计
- 可降级

## 4.6 Execution Layer

职责：

- 把通过审批的决策转换成真实执行

核心模块：

- Trading Gateway
- Order State Machine
- Reconciliation Engine
- Position Engine
- Emergency Kill Switch

要求：

- 高可靠
- 强幂等
- 强对账
- 强风险优先级

## 5. 插件化蓝图

## 5.1 插件类型

### Data Plugin

负责：

- 行情
- 事件
- 社交数据
- 链上数据

### Feature Plugin

负责：

- 技术特征
- 结构特征
- 资金特征
- 情绪特征

### Skill Plugin

负责：

- 入场
- 加仓
- 减仓
- 退出
- 风险收缩

### Agent Plugin

负责：

- Research Agent
- Debate Agent
- Review Agent
- Explain Agent

### Risk Plugin

负责：

- 风险限制
- 熔断规则
- 事件级风险策略

### UI Plugin

负责：

- 新报表
- 新面板
- 新工作台视图

## 5.2 插件生命周期

每个插件统一经历：

1. `registered`
2. `validated`
3. `sandboxed`
4. `approved`
5. `active`
6. `deprecated`
7. `archived`

## 5.3 插件准入门槛

必须满足：

- License 可接受
- 依赖可审查
- 有版本号
- 有边界说明
- 可回滚
- 可审计
- 可在沙箱中验证

## 6. 持续导入开源模块的长期机制

## 6.1 开源雷达

建立固定跟踪池：

- 交易框架池
- 数据接入池
- Agent / Workflow 池
- RAG / Graph 池
- 风控 / 组合池
- Binance Futures 专项池

## 6.2 固定节奏

建议制度：

- 每周做一次增量扫描
- 每月做一次重点项目评审
- 每季度做一次引入/淘汰决策

## 6.3 仓库评分维度

统一按以下维度评分：

- 目标贴合度
- 架构可插拔性
- 社区活跃度
- 维护稳定性
- License 风险
- 嵌入成本
- 替代难度
- 生产可控性

## 7. 3 年演进路线

## 第一年：打基础

目标：

- 建协议
- 建标准 Adapter
- 建 Skill Registry
- 建 Risk Engine
- 建研究层

重点：

- 先具备吸收外部能力的能力

## 第二年：做增强

目标：

- 大规模导入外部研究能力
- 形成知识层和图谱层
- 构建多 Agent 研究闭环

重点：

- 从“接入能力”升级为“消化能力”

## 第三年：做生态

目标：

- Skill 市场化
- Connector 市场化
- 报告与 Agent 模板市场化

重点：

- 从内部平台升级为生态化平台

## 8. 关键制度

### 升级制度

外部模块升级必须走：

- 镜像
- 沙箱
- 回归
- 灰度
- 正式替换

### 退出制度

任何外部模块都必须可移除，不能把平台绑死在某个开源依赖上。

### 审计制度

所有外部模块接入都必须保留：

- 版本号
- 调用范围
- 数据来源
- 风险边界
- 影响面

## 9. 最终目标形态

AICrypto 的理想终局应当是：

- 内核稳定
- 插件丰富
- 开源能力持续输入
- 风控与执行始终自主可控
- 研究、知识、事件、图谱和策略持续进化

这意味着 AICrypto 长期不是“某个开源仓库的封装壳”，而是一个可以持续吸收全球开源创新成果的 AI 加密货币自动化投研交易平台。
