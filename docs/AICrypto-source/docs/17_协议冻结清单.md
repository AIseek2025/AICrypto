# AICrypto 协议冻结清单

## 1. 文档目标

本文件用于冻结 AICrypto 首版工程启动前必须稳定的核心协议。

这里的“冻结”不是说永远不能改，而是指：

- 进入工程初始化后，默认按本清单实施
- 若后续修改，必须走版本升级和兼容评审
- 任何外部开源模块接入时，都必须先对齐这些协议

这份文档的意义在于，先把平台最稳定的骨骼定下来，再开始做模块、目录和工程初始化。

## 2. 冻结原则

### 2.1 少而稳

首版只冻结真正必要的核心协议，不追求一次把所有对象都定义到极致。

### 2.2 先跨模块，再细实现

优先冻结那些会跨越多个层级、多个服务、多个仓库边界的协议。

### 2.3 显式版本化

所有协议首版统一定义：

- `schema_name`
- `schema_version`
- `producer`
- `created_at`
- `trace_id`

## 3. 首版必须冻结的协议

## 3.1 `CanonicalEvent`

用途：

- 平台内部统一事件总线对象

来源：

- 行情
- 交易所用户流
- 社交事件
- 新闻事件
- 链上事件
- 系统内部状态事件

首版冻结字段：

- `schema_name`
- `schema_version`
- `event_id`
- `trace_id`
- `source_type`
- `source_name`
- `event_type`
- `symbol`
- `ts_event`
- `ts_ingested`
- `payload`
- `quality_flags`
- `tags`

冻结约束：

- 所有外部输入必须先标准化成 `CanonicalEvent`
- `payload` 内允许保留 source-specific 字段，但顶层结构必须统一

## 3.2 `MarketSnapshot`

用途：

- 表示某个时点某标的的市场快照

首版冻结字段：

- `schema_name`
- `schema_version`
- `symbol`
- `exchange`
- `market_type`
- `last_price`
- `mark_price`
- `index_price`
- `funding_rate`
- `open_interest`
- `volume_24h`
- `ts_snapshot`

冻结约束：

- 用于特征计算和状态判断
- 不直接替代原始逐笔流

## 3.3 `FeatureVector`

用途：

- 承载统一特征输出

首版冻结字段：

- `schema_name`
- `schema_version`
- `feature_set`
- `feature_version`
- `symbol`
- `window`
- `ts_feature`
- `features`
- `source_refs`

冻结约束：

- 不要求首版冻结所有特征字段名
- 但必须冻结特征集合的元数据结构

## 3.4 `StrategySpec`

用途：

- 描述策略对象本身

首版冻结字段：

- `strategy_id`
- `strategy_name`
- `strategy_type`
- `owner`
- `input_requirements`
- `signal_model`
- `risk_assumptions`
- `execution_constraints`
- `status`
- `version`

冻结约束：

- `StrategySpec` 表示“策略”
- 与 `SkillSpec` 明确区分

## 3.5 `SkillSpec`

用途：

- 描述可被 Agent 或 Strategy 调用的标准交易动作单元

首版冻结字段：

- `skill_id`
- `skill_name`
- `skill_family`
- `entry_conditions`
- `position_rules`
- `add_rules`
- `reduce_rules`
- `exit_rules`
- `risk_rules`
- `applicable_market_states`
- `input_contract`
- `output_contract`
- `status`
- `version`

冻结约束：

- 首版必须支持“主升浪、主跌浪、相关性事件、风险收缩”四类 Skill

## 3.6 `SignalEvent`

用途：

- 表示研究或模型产生的候选信号

首版冻结字段：

- `signal_id`
- `signal_type`
- `symbol`
- `direction`
- `confidence`
- `horizon`
- `reason_codes`
- `evidence_refs`
- `ts_signal`

冻结约束：

- `SignalEvent` 不是订单
- 必须继续流向 `Decision` 与 `Risk` 层处理

## 3.7 `OrderIntent`

用途：

- 表示通过决策层和风控前置后的候选交易意图

首版冻结字段：

- `intent_id`
- `account_scope`
- `symbol`
- `side`
- `position_side`
- `order_type`
- `quantity`
- `price_limit`
- `reduce_only`
- `leverage_hint`
- `take_profit_hint`
- `stop_loss_hint`
- `time_in_force`
- `origin_ref`
- `ts_intent`

冻结约束：

- 这是进入执行层的标准入口
- 执行层不得直接消费原始 `SignalEvent`

## 3.8 `RiskDecision`

用途：

- 表示统一风险网关的裁决结果

首版冻结字段：

- `decision_id`
- `target_ref`
- `decision`
- `severity`
- `rule_hits`
- `exposure_snapshot`
- `required_actions`
- `review_required`
- `ts_decision`

冻结约束：

- `decision` 至少包含：
  - `allow`
  - `deny`
  - `review`
  - `shrink`

## 3.9 `ExecutionReport`

用途：

- 表示执行层反馈

首版冻结字段：

- `report_id`
- `intent_id`
- `exchange`
- `symbol`
- `order_status`
- `filled_qty`
- `avg_fill_price`
- `fees`
- `exchange_order_id`
- `raw_status`
- `reconcile_state`
- `ts_report`

冻结约束：

- 所有下单结果、撤单结果、部分成交和补偿结果，都必须最终映射到 `ExecutionReport`

## 3.10 `ReviewReport`

用途：

- 表示研究复盘或 Agent 审查的标准结果对象

首版冻结字段：

- `review_id`
- `review_type`
- `target_ref`
- `summary`
- `findings`
- `recommendations`
- `evidence_refs`
- `reviewer`
- `ts_review`

冻结约束：

- 用于研究闭环和知识沉淀
- 不直接进入执行层

## 4. 首版不冻结的内容

以下内容首版先不做过度冻结，只保留设计空间：

- 全量特征字段命名
- 全量图谱节点/边结构
- 全量 UI 组件协议
- 多账户清算细节
- 跨交易所统一订单细节的全部字段

原因：

- 这些内容仍需在后续沙箱和模块落地中迭代

## 5. 版本规则

首版统一采用：

- `v1`

后续升级原则：

- 增加可选字段：允许小版本升级
- 修改语义或删除字段：必须大版本升级
- 跨模块影响：必须同步更新引用清单

## 6. 冻结后的工程纪律

从本文件开始，后续所有文档和初始化工程都必须遵守：

- 新模块先声明使用哪些协议
- 新开源仓库先判断映射到哪些协议
- 新功能先判断是否要扩展现有协议，而不是随意新建对象

## 7. 最终结论

AICrypto 首版真正要保护的不是某个实现，而是这组核心协议。

只要协议稳定：

- 目录可以演进
- 实现可以替换
- 开源模块可以增删
- 技术栈可以调整

但平台仍然不会失控。
