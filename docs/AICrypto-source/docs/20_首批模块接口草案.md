# AICrypto 首批模块接口草案

## 1. 文档目标

本文件用于定义首批核心模块的接口草案。

这里的“接口”不是具体代码语法，而是工程层面的模块契约，解决：

- 模块对外提供什么能力
- 模块接受什么输入
- 模块输出什么结果
- 模块依赖什么协议
- 模块不负责什么

## 2. 首批接口设计原则

### 2.1 输入输出必须协议化

所有首批模块都必须直接引用：

- `CanonicalEvent`
- `MarketSnapshot`
- `FeatureVector`
- `SkillSpec`
- `SignalEvent`
- `OrderIntent`
- `RiskDecision`
- `ExecutionReport`
- `ReviewReport`

### 2.2 模块只做一层职责

例如：

- Adapter 只做接入与映射
- Intelligence 只做分析与候选生成
- Decision 只做约束决策
- Execution 只做执行与反馈

### 2.3 先模块契约，后实现细节

首版先明确模块边界，不急着定义全部内部函数。

## 3. 首批核心模块接口

## 3.1 Binance Market Data Adapter

模块定位：

- 对接 Binance Futures 市场数据流

模块路径建议：

- `adapters/exchanges/binance/market_data`

输入：

- Binance WebSocket 原始消息
- Binance REST 历史数据响应

输出：

- `CanonicalEvent`
- `MarketSnapshot`

核心接口草案：

- `connect_streams(config) -> stream_handle`
- `backfill(range, symbols) -> raw_records`
- `normalize(raw_event) -> CanonicalEvent`
- `build_snapshot(events) -> MarketSnapshot`
- `health() -> adapter_status`

不负责：

- 策略判断
- 风控裁决
- 正式下单

## 3.2 Feature Engine

模块定位：

- 从市场数据和事件流中提取标准化特征

模块路径建议：

- `intelligence/features`

输入：

- `CanonicalEvent`
- `MarketSnapshot`

输出：

- `FeatureVector`

核心接口草案：

- `register_feature_set(feature_set) -> ack`
- `compute(snapshot, context) -> FeatureVector`
- `batch_compute(range, symbols) -> list[FeatureVector]`
- `validate(feature_vector) -> validation_result`

不负责：

- 直接发出交易意图

## 3.3 Market State Engine

模块定位：

- 判断市场状态

模块路径建议：

- `intelligence/states`

输入：

- `FeatureVector`
- `CanonicalEvent`

输出：

- `SignalEvent`
- `state_label`

核心接口草案：

- `evaluate(symbol, features, context) -> state_result`
- `classify_regime(state_result) -> regime_label`
- `emit_signal(state_result) -> SignalEvent | None`

不负责：

- 直接生成订单

## 3.4 Correlation Engine

模块定位：

- 处理人物、赛道、BTC、板块、情绪和事件对币种的相关性影响

模块路径建议：

- `intelligence/correlations`

输入：

- `CanonicalEvent`
- `FeatureVector`
- 事件上下文

输出：

- `SignalEvent`
- `reason_codes`

核心接口草案：

- `ingest_event(event) -> ack`
- `score_relations(symbol, context) -> correlation_score`
- `generate_signal(score, context) -> SignalEvent | None`

不负责：

- 仓位规划
- 风控决策

## 3.5 Backtest Lab

模块定位：

- 回测与实验环境

模块路径建议：

- `intelligence/backtest`
- `research/experiments`

输入：

- `StrategySpec`
- `SkillSpec`
- 历史数据集

输出：

- 回测结果
- 证据报告
- 复盘素材

核心接口草案：

- `run_backtest(strategy, dataset, config) -> backtest_result`
- `evaluate(result) -> metrics`
- `render_report(result) -> report_ref`
- `promote_candidate(result) -> proposal`

不负责：

- 生产实盘执行

## 3.6 Knowledge Service

模块定位：

- 统一知识对象、研究档案、Skill 文档和复盘材料

模块路径建议：

- `intelligence/knowledge`

输入：

- 研究报告
- 回测报告
- `ReviewReport`
- `SkillSpec`

输出：

- knowledge object
- 检索结果
- 引用结果

核心接口草案：

- `index(document) -> doc_ref`
- `search(query, scope) -> search_results`
- `fetch(ref) -> knowledge_object`
- `link_skill(skill_id, refs) -> ack`

不负责：

- 实盘执行

## 3.7 Research Agent Graph

模块定位：

- 组织研究型 Agent 流程

模块路径建议：

- `intelligence/agents/research_graph`

输入：

- 问题
- 数据引用
- 知识引用

输出：

- `ReviewReport`
- `SignalEvent`
- 候选策略建议

核心接口草案：

- `start(task, context) -> run_id`
- `step(run_id, input) -> state`
- `finalize(run_id) -> ReviewReport`

不负责：

- 绕过风控直接交易

## 3.8 Skill Registry

模块定位：

- 管理 Skill 资产

模块路径建议：

- `decision/skill_router`
- 也可单独抽象为 `skills registry service`

输入：

- `SkillSpec`
- 技能元数据

输出：

- skill lookup result
- skill selection result

核心接口草案：

- `register(skill_spec) -> skill_id`
- `get(skill_id) -> SkillSpec`
- `find(criteria) -> list[SkillSpec]`
- `resolve(signal, market_state) -> list[SkillSpec]`

不负责：

- 最终执行

## 3.9 Risk Gateway

模块定位：

- 统一交易风险闸门

模块路径建议：

- `decision/risk_gateway`

输入：

- `SignalEvent`
- `OrderIntent`
- 账户暴露
- 风险规则集

输出：

- `RiskDecision`

核心接口草案：

- `pre_trade(intent, context) -> RiskDecision`
- `in_trade(position_state, context) -> RiskDecision`
- `post_trade(execution_report, context) -> RiskDecision`
- `explain(decision_id) -> explanation`

不负责：

- 直接发单

## 3.10 Execution Planner

模块定位：

- 把通过风控的候选动作细化为执行意图

模块路径建议：

- `decision/planner`

输入：

- `SignalEvent`
- `SkillSpec`
- `RiskDecision`

输出：

- `OrderIntent`

核心接口草案：

- `plan(signal, skill, context) -> OrderIntent`
- `adjust(intent, risk_decision) -> OrderIntent`

不负责：

- 直接与交易所通信

## 3.11 Trading Gateway

模块定位：

- 对接交易所下单与回执

模块路径建议：

- `execution/gateway`

输入：

- `OrderIntent`

输出：

- `ExecutionReport`

核心接口草案：

- `submit(intent) -> ExecutionReport`
- `cancel(order_ref) -> ExecutionReport`
- `query(order_ref) -> ExecutionReport`

不负责：

- 自己决定该不该交易

## 3.12 Order State Machine

模块定位：

- 维护订单生命周期

模块路径建议：

- `execution/order_state`

输入：

- `OrderIntent`
- 交易所状态回报
- 对账结果

输出：

- 标准订单状态
- 状态迁移记录

核心接口草案：

- `init(intent) -> order_state`
- `transition(current_state, event) -> new_state`
- `reconcile(order_ref, exchange_data) -> reconcile_result`

不负责：

- 交易决策

## 3.13 Reconciliation Engine

模块定位：

- 对账与异常补偿

模块路径建议：

- `execution/reconciliation`

输入：

- 交易所订单状态
- 本地订单状态
- `ExecutionReport`

输出：

- reconcile result
- 补偿动作建议

核心接口草案：

- `compare(local, remote) -> diff`
- `repair(diff) -> repair_plan`
- `confirm(repair_plan) -> final_status`

## 3.14 Review Archive

模块定位：

- 复盘归档和知识回流

模块路径建议：

- `intelligence/knowledge/review_archive`

输入：

- `ReviewReport`
- 回测结论
- 执行结果

输出：

- 归档条目
- 检索引用

核心接口草案：

- `archive(review_report) -> archive_ref`
- `query(filters) -> archive_results`
- `link_execution(review_ref, execution_refs) -> ack`

## 4. 模块依赖关系

建议首版依赖链如下：

1. `Binance Market Data Adapter`
2. `Feature Engine`
3. `Market State Engine / Correlation Engine`
4. `Skill Registry`
5. `Execution Planner`
6. `Risk Gateway`
7. `Trading Gateway`
8. `Order State Machine`
9. `Reconciliation Engine`

知识层和 Agent 层并行支持：

- `Knowledge Service`
- `Research Agent Graph`
- `Review Archive`

## 5. 首批接口冻结范围

首版建议冻结的是：

- 模块职责
- 核心输入输出
- 所依赖协议
- 明确不负责的事项

首版暂不冻结：

- 具体编程语言语法
- 具体类名
- 具体 HTTP / RPC / async 细节

## 6. 最终结论

这份接口草案的作用是：

- 让目录骨架不是空壳
- 让后续工程初始化时，每个模块一开始就有边界
- 让开源仓库嫁接时，知道它是在补哪个接口，而不是随意贴进来

这一步完成后，AICrypto 已经具备“随时进入首轮工程骨架搭建”的条件。
