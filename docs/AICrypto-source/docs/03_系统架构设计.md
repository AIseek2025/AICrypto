# AICrypto 系统架构设计

## 1. 架构目标

系统架构必须同时满足以下要求：

- 支持币安全市场永续合约的持续数据采集与特征计算。
- 支持研究、回测、仿真、实盘四个环境的一致运行。
- 支持多 Agent、多 Skill、多策略组合的自动调度。
- 支持强风控、强审计、强可观测性。
- 核心热路径能向高并发 Rust 架构平滑演进。

## 2. 总体分层

建议采用 7 层架构：

1. 接入层
   - 币安 API Adapter
   - 外部事件源 Adapter
2. 数据层
   - 原始数据采集
   - 标准化事件总线
   - 时序与对象存储
3. 特征层
   - 技术特征
   - 资金与结构特征
   - 相关性特征
   - 事件特征
4. 研究与策略层
   - 回测引擎
   - 策略实验工坊
   - Skill Registry
5. 决策层
   - Market State Engine
   - Signal Engine
   - Portfolio Optimizer
   - Agent Orchestrator
6. 执行与风控层
   - Risk Gateway
   - Execution Router
   - Order State Machine
   - Reconciliation
7. 产品与运营层
   - 控制台
   - 监控
   - 报告
   - 管理后台

## 3. 服务拆分建议

### 3.1 Rust 核心服务

1. `gateway-market`
   - 交易所市场数据接入
2. `gateway-trading`
   - 统一交易执行网关
3. `risk-engine`
   - 实时风控
4. `signal-runtime`
   - 实时信号评估
5. `portfolio-engine`
   - 组合仓位与暴露控制
6. `event-bus-consumer`
   - 实时特征消费与处理
7. `reconciliation-service`
   - 对账与异常修复

### 3.2 Python 研究服务

1. `research-notebook`
   - 因子探索与事件分析
2. `strategy-lab`
   - 规则研究、样本分析、参数优化
3. `model-lab`
   - 分类、排序、概率预测模型实验
4. `feature-offline-pipeline`
   - 离线特征回补与训练样本生成

### 3.3 TypeScript 产品层

1. `console-web`
   - 运营控制台
2. `admin-api`
   - 后台权限、配置、审计
3. `report-center`
   - 报告与复盘页面

## 4. 核心数据流

### 4.1 市场数据流

`Binance REST/WS -> Adapter -> Canonical Events -> NATS/Redpanda -> Feature Engine -> TimescaleDB/Object Storage -> Signal Engine`

### 4.2 交易数据流

`Signal/Skill -> Risk Engine -> Execution Router -> Binance -> User Stream -> Order State Machine -> Portfolio Engine -> Console`

### 4.3 研究闭环

`Raw Data -> Feature Warehouse -> Research Lab -> Strategy Spec -> Skill Package -> Backtest -> Paper Trading -> Canary Live`

## 6. 存储架构

### 6.1 PostgreSQL + TimescaleDB

适合保存：

- 合约元数据
- 时序特征
- 信号记录
- 订单与仓位
- 审计日志
- 策略配置
- Skill Registry

### 6.2 Redis

适合保存：

- 实时热数据缓存
- 最近行情状态
- 风控限额
- 会话与权限缓存
- 幂等键

### 6.3 对象存储

适合保存：

- 原始事件归档
- 回测结果
- 策略报告
- 模型文件
- 训练样本快照

## 8. 风控架构

风控必须是独立服务，位于信号之后、执行之前，并拥有最终 veto 权。

### 8.1 风控检查层级

1. 账户级
   - 最大总仓位
   - 最大日亏损
   - 最大回撤
   - 最大杠杆
2. 标的级
   - 单币最大暴露
   - 最低流动性门槛
   - 最大滑点阈值
   - 禁开仓窗口
3. 策略级
   - 最大连续亏损次数
   - 最大并发信号
   - 策略冷却时间
4. 组合级
   - 高相关币种聚集暴露
   - 同板块集中暴露
   - 多空失衡
5. 系统级
   - 接口异常率过高
   - 用户流断开
   - 行情延迟超阈值
   - 对账失败

### 8.2 熔断策略

- 只禁止开新仓
- 自动降低仓位
- 仅允许 reduce-only
- 全局紧急平仓
- 切换到人工审批模式

## 9. 多环境体系

必须至少有 4 套逻辑环境：

1. `dev`
   - 开发联调
2. `backtest`
   - 历史回测
3. `paper`
   - 实时仿真
4. `prod`
   - 实盘

要求同一 Skill 在四套环境中的接口一致，只替换行情源、订单执行器和风控阈值。

## 10. 可观测性设计

### 10.1 Metrics

- 数据接入延迟
- Stream 重连次数
- 特征计算耗时
- 信号触发数量
- 风控拦截数量
- 下单成功率
- ACK 延迟
- 成交率
- 滑点
- 实时 PnL
- 未知状态订单数量

### 10.2 Logs

- 市场接入日志
- 订单审计日志
- 风控决策日志
- 策略推理日志
- Agent 调度日志

### 10.3 Tracing

每次交易从“信号出现 -> 风控审批 -> 下单 -> 回报 -> 平仓”必须一条 trace 可追溯。

## 11. 安全与权限

- 交易密钥与研究环境彻底隔离。
- 关键操作支持双人审批。
- 管理后台按 RBAC 控制权限。
- 所有配置改动与策略上架下架都需要审计。
- 外部事件源要做可信度评级，防止被伪造消息诱导交易。

## 12. 推荐部署形态

### 12.1 首版

- 单 VPC
- 研究服务与生产服务分命名空间
- PostgreSQL 主实例
- Redis 主从
- NATS JetStream
- 核心 Rust 服务容器化部署

### 12.2 成熟版

- 多可用区部署
- 分区消息总线
- 多账户执行路由
- 主从或双活对账中心
- 独立风险集群

## 13. 为什么核心热路径必须 Rust

AICrypto 在成熟阶段会同时面对：

- 全市场多标的高频行情流
- 高频特征计算
- 多策略并发评估
- 严格风控判断
- 低延迟下单与状态机驱动

这些都属于高并发、低延迟、强可靠场景，适合 Rust 作为核心底座。Python 继续用于研究生产力，但不宜承担最终生产热路径。
