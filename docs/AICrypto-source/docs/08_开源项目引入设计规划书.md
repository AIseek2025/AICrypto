# AICrypto 开源项目引入设计规划书

## 1. 目标

本规划书用于回答三个问题：

- 哪些开源项目适合 AICrypto。
- 这些项目应该如何被嫁接，而不是粗暴拼装。
- 平台如何建立长期、持续、可治理的开源引入机制。

核心原则不是“把外部仓库直接拿来跑”，而是把优秀能力拆解、协议化、模块化，再纳入 AICrypto 的平台体系。

## 2. 引入总原则

### 2.1 平台标准先于仓库实现

外部仓库只能提供能力，不得反向定义 AICrypto 的核心领域模型。AICrypto 必须先坚持自己的标准：

- Canonical Event 协议
- Strategy Spec / Skill Spec
- Risk Policy Spec
- Order Intent / Order State Model
- Audit Trace 模型

任何外部项目进入平台前，都必须先适配这些标准。

### 2.2 热路径内生，外围能力外接

应当长期内生的能力：

- Binance 正式 Adapter
- Risk Engine
- Execution Router
- Order State Machine
- Position / PnL Engine
- 审计链路

可以长期借助开源生态的能力：

- 回测与研究工具
- AI Agent 编排
- RAG 与知识检索
- 组合优化
- 图谱与事件理解
- 数据探索与报告生成

### 2.3 五种引入方式

1. 镜像归档
2. 代码参考
3. 沙箱旁路运行
4. 模块能力抽取
5. 协议级兼容

## 3. 项目分层筛选

## 3.1 A 类：建议重点引入并长期跟踪

### Freqtrade

定位：

- 加密交易机器人
- 回测
- 参数优化
- 策略实验平台

适配价值：

- 适合快速建立研究与策略实验工坊
- 带有成熟的 dry-run、回测、配置、策略组织方式
- 对 Futures 有现实参考意义

建议接入方式：

- 作为研究/回测参考仓库
- 借鉴策略接口、回测流程、参数搜索流程
- 不作为 AICrypto 正式交易内核

### NautilusTrader

定位：

- 高性能事件驱动量化平台
- 强 research-to-live parity
- 偏专业化基础设施

适配价值：

- 非常适合参考执行层、状态机、适配器架构
- 与 AICrypto 长期 Rust-first 方向高度一致
- 是未来高质量执行架构的重要参考样板

建议接入方式：

- 重点研究其事件模型、实盘/回测一致性、订单生命周期
- 不整体照搬
- 作为 AICrypto 执行层设计标杆

### Cryptofeed

定位：

- 统一加密市场数据流接入层

适配价值：

- 支持 Binance Futures
- 支持 trades、book、funding、open interest、liquidations 等关键流
- 适合快速建立行情接入沙箱

建议接入方式：

- 可在研究环境直接试跑
- 生产环境逐步沉淀为 AICrypto 自己的正式 Adapter
- 借鉴其标准化思路和 callback/backends 结构

### Riskfolio-Lib

定位：

- 组合优化与风险预算研究库

适配价值：

- 适合后续多策略组合和资金分配
- 可以支持 Portfolio Lab 的研究层能力

建议接入方式：

- 放入 Python 研究层
- 输出资金分配建议与组合风险预算
- 不直接承担交易前毫秒级风险拦截

### LangGraph

定位：

- 有状态、多步骤、多 Agent 编排框架

适配价值：

- 适合 Research Agent、Review Agent、Debate Agent
- 适合人机协同审批、持久化、分支和循环工作流

建议接入方式：

- 作为 Agent 编排层首选方案
- 只进入研究、复盘、审批和解释链路
- 不直接获得实盘最终下单权

### LlamaIndex

定位：

- RAG 与知识访问框架

适配价值：

- 适合接入研究报告、Skill 文档、回测结果、事件档案
- 适合形成 Skill Knowledge Base 与 Research Memory

建议接入方式：

- 作为知识层与检索层引擎
- 配合 `pgvector` 或后续 `Qdrant`
- 不进入执行热路径

## 3.2 B 类：选择性引入

### TradingAgents

价值：

- 多角色投研辩论机制有参考价值
- 适合借鉴 analyst / trader / risk manager 分工

限制：

- 更偏研究框架
- 原生股票研究味更重
- 不适合直接做币安永续生产主干

建议：

- 借鉴角色设计和研究报告结构
- 不直接作为平台中枢

### ai-auto-trader-ahh

价值：

- 更贴近 Binance Futures 自动交易实践
- 有多 AI 角色、止盈止损、用户流、风控规则等实战参考

限制：

- 架构更偏单项目
- 更适合作为战术参考而不是平台底座

建议：

- 抽取其 Binance 永续实战逻辑与风险保护经验
- 沉淀为 AICrypto 的 Skill 或 Risk Policy

### Jesse / Backtrader / vn.py

价值：

- 适合原型验证、回测参考、中文团队认知迁移

限制：

- 平台化、现代数据工程、多 Agent、长期可插拔能力不足

建议：

- 保留为专项参考，不进入主线底座

## 3.3 C 类：持续观察但暂不集成

- Hummingbot：适合做市/套利，不是主投研平台底座
- Lean：研究体系完整，但币安永续贴合度不够高
- CrewAI / AutoGen：适合 PoC，不是当前首选核心编排
- 各类单一“AI 预测价格”仓库：大多只适合作为案例参考

## 4. 嫁接设计

## 4.1 三层嫁接结构

### 第一层：研究层嫁接

目标：

- 提升研究效率
- 快速形成策略实验能力

候选：

- Freqtrade
- Riskfolio-Lib
- LangGraph
- LlamaIndex

### 第二层：数据与事件层嫁接

目标：

- 快速扩大可观测能力
- 提升事件驱动与相关性研究能力

候选：

- Cryptofeed
- LlamaIndex
- 后续图谱工具

### 第三层：执行层参考嫁接

目标：

- 强化实盘基础设施设计

候选：

- NautilusTrader
- ai-auto-trader-ahh

说明：

- 这一层优先“参考与吸收”，不建议直接整仓库嵌入主热路径

## 4.2 统一接入流程

任何开源仓库都必须经过以下流程：

1. 扫描与收集
2. License 审查
3. 活跃度评估
4. 本地镜像归档
5. 沙箱验证
6. 能力拆解
7. 协议适配
8. 小范围嫁接
9. 回归测试
10. 正式纳入平台

## 5. 本地优先归档清单

当前首批建议长期跟踪并已优先归档的仓库：

- `freqtrade`
- `nautilus_trader`
- `cryptofeed`
- `Riskfolio-Lib`
- `TradingAgents`
- `ai-auto-trader-ahh`
- `langgraph`
- `llama_index`

## 6. 风险边界

### 不允许的接入方式

- 外部 Agent 直接获得生产交易密钥
- 外部仓库直接写入 AICrypto 核心数据库
- 外部项目绕过 Risk Engine
- 外部仓库的 symbol / order / position 模型直接污染平台标准

### 必须坚持的约束

- 所有交易意图先过 AICrypto Risk Gateway
- 所有外部数据先过 Canonical Event 标准化
- 所有外部策略先转成 Skill Spec 或 Strategy Spec
- 所有外部模块必须可审计、可降级、可移除

## 7. 执行建议

短期优先级：

1. 用 `Freqtrade + Riskfolio-Lib` 强化研究层
2. 用 `Cryptofeed` 强化行情侧沙箱
3. 用 `LangGraph + LlamaIndex` 搭建研究智能体与知识层
4. 用 `NautilusTrader + ai-auto-trader-ahh` 反向完善执行和风控设计

中期优先级：

1. 建立开源仓库评分机制
2. 建立统一升级与淘汰机制
3. 建立持续跟踪池与月度评审制度

## 8. 结论

AICrypto 的正确路线不是“用某个开源项目替代自己”，而是：

- 用开源项目加速研究
- 用开源项目校准架构
- 用开源项目补齐专项能力
- 最终把核心能力沉淀为 AICrypto 自己的平台标准和正式模块
