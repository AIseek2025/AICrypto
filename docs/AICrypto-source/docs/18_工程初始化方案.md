# AICrypto 工程初始化方案

## 1. 文档目标

本文件用于把前面的总纲、架构、开源调研、模块规划、协议冻结，进一步落到“正式工程如何启动”。

目标是明确：

- 工程初始化的目标状态
- 初始化顺序
- 首批目录和仓库骨架
- 第一轮不该做什么

本文件是 AICrypto 从文档规划阶段进入正式工程阶段的桥梁。

## 2. 初始化总原则

### 2.1 先搭平台骨架，不先写业务细节

第一轮初始化只做三件事：

- 搭结构
- 定协议
- 建边界

不应在初始化阶段就着急堆策略逻辑和产品页面。

### 2.2 先 monorepo，后模块扩展

首版建议采用单仓多模块方式，原因：

- 当前平台还在快速演进
- 需要频繁调整协议和模块边界
- 开源仓库嫁接需要统一治理

### 2.3 研究与生产分区

初始化阶段必须从目录层面就分开：

- 研究实验空间
- 正式平台空间

这样后续引入开源项目时，研究层不会污染正式生产层。

## 3. 初始化目标状态

第一轮初始化完成后，仓库应达到以下状态：

- 有明确 monorepo 目录
- 有协议目录
- 有 Adapter 壳
- 有 Intelligence 壳
- 有 Decision 壳
- 有 Execution 壳
- 有 Research 区
- 有 Skills 区
- 有 External 镜像区
- 有 Docs 索引闭环

## 5. 初始化阶段的目录职责

### `apps/`

用途：

- Web console
- Admin API
- Ops tools

初始化阶段：

- 只建立壳，不做复杂 UI

### `foundation/`

用途：

- 配置
- 日志
- 监控
- 安全
- 存储抽象
- 消息总线抽象

初始化阶段：

- 先建立统一命名和边界文档

### `protocols/`

用途：

- 存放首版冻结协议定义

初始化阶段：

- 这是最优先建立的目录

### `adapters/`

用途：

- 对接 Binance 和后续外部数据源

初始化阶段：

- 先建立 `binance` 目录和数据接入壳

### `intelligence/`

用途：

- 特征、状态、图谱、知识、Agent、回测

初始化阶段：

- 先建子目录和 README，不急着实现全部逻辑

### `decision/`

用途：

- Skill Router
- Capital Allocator
- Planner
- Risk Gateway

初始化阶段：

- 先把风控和决策边界从目录上分清

### `execution/`

用途：

- Gateway
- Order State
- Positions
- Reconciliation
- Emergency

初始化阶段：

- 先以架构壳和状态机文档为主

### `research/`

用途：

- notebooks
- experiments
- datasets
- reports

初始化阶段：

- 这里是开源仓库借力和实验的主战场

### `skills/`

用途：

- 平台正式 Skill 资产库

初始化阶段：

- 先按家族分类建目录

### `infra/`

用途：

- 本地开发环境
- 部署说明
- 基础设施模板

### `tests/`

用途：

- 协议测试
- 适配器测试
- 回放测试
- 风控规则测试

## 6. 初始化顺序

建议分 4 个小阶段。

## 阶段 1：冻结骨架

任务：

- 建立目录
- 建立顶层 README
- 建立 `protocols`
- 建立每层 README

目标：

- 任何后续模块都有位置可落

## 阶段 2：冻结协议

任务：

- 把 `17_协议冻结清单.md` 对应到目录结构
- 为每个核心协议建立文档或 schema 占位

目标：

- 后续实现不再各自定义对象

## 阶段 3：建立沙箱边界

任务：

- 明确 `research` 和正式模块的隔离
- 为 `external/open_source` 建立引用说明
- 为 `adapters/binance`、`intelligence/knowledge`、`intelligence/agents` 建立壳

目标：

- 让开源项目可以开始被挂接到正确层次

## 阶段 4：建立实施入口

任务：

- 明确首批模块 owner
- 明确先后顺序
- 明确第一轮编码入口

目标：

- 为下一步正式启动编码提供清晰入口

## 7. 首批初始化优先目录

第一轮最建议优先建立的目录是：

- `protocols/canonical_event`
- `protocols/skill_spec`
- `protocols/order_intent`
- `protocols/risk_decision`
- `adapters/exchanges/binance`
- `intelligence/knowledge`
- `intelligence/agents`
- `decision/risk_gateway`
- `execution/order_state`
- `research/experiments`
- `skills/trend`
- `skills/correlation`

## 8. 初始化阶段不该做什么

### 不要过早做复杂前端

原因：

- 前端会快速变化
- 当前最重要的是底层边界与协议

### 不要过早写大量策略代码

原因：

- Skill 协议尚未真正落目录
- 过早写策略容易返工

### 不要过早绑定单一开源框架

原因：

- AICrypto 的目标是平台化，不是某个框架的二次封装

### 不要把研究脚本直接放进生产目录

原因：

- 会破坏长期治理

## 9. 首批工程启动建议

如果下一步开始编码，建议启动顺序如下：

1. 初始化 monorepo 目录骨架
2. 创建协议目录和协议 README / schema 占位
3. 创建 Binance Adapter 壳
4. 创建 Knowledge / Agents / Risk Gateway / Order State 壳
5. 创建 `research` 与 `skills` 目录
6. 建立 `external/open_source` 引用规则

## 10. 代码层建议

虽然这一步不编码，但工程初始化时建议语言分工保持一致：

- `Rust`
  - 协议实现候选
  - 数据热路径
  - 执行和风控热路径
- `Python`
  - research
  - RAG
  - agents
  - 回测和实验
- `TypeScript`
  - console
  - admin
  - 工具层

## 11. 工程初始化的完成标志

以下条件满足后，可以认为工程初始化完成：

- 目录结构稳定
- 协议目录到位
- 每层职责清楚
- 研究/生产隔离清楚
- 开源仓库归档区清楚
- 第一轮模块入口清楚

## 12. 最终结论

AICrypto 的工程初始化，不是搭一个空项目这么简单，而是：

- 把未来 1-3 年的模块治理方式先搭好
- 把开源能力导入路径先搭好
- 把协议和边界先搭好

只要这一步扎实，后续正式编码就会稳定得多。
